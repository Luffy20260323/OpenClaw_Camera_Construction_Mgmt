一、核心设计原则

1.1 单一权限源（Single Source of Truth）

┌─────────────────────────────────────────────────────────┐ │ 权限唯一来源 │ │ permissions 表 │ │ │ │ permission_code (唯一标识) │ │ permission_name (中文名称) │ │ permission_type (API | UI | DATA) │ │ resource_type (用户 | 角色 | 作业区 | 菜单...) │ │ action (create | read | update | delete | export...) │ └─────────────────────────────────────────────────────────┘
核心原则：
  • ✅ 每个权限有唯一的 permission_code
  • ✅ 后端 API 权限是唯一验证点
  • ✅ 前端 UI 权限从 API 权限推导
  • ✅ 数据权限通过注解自动注入

1.2 权限分层模型

┌──────────────────────────────────────────────────────────┐ │ 权限分层架构 │ ├──────────────────────────────────────────────────────────┤ │ │ │ L1: 功能权限 (Functional Permission) │ │ └── 能否访问某个功能/API │ │ └── 例：user:create, workarea:view │ │ │ │ L2: 数据权限 (Data Permission) │ │ └── 能否访问某条数据 │ │ └── 例：只能看本公司的作业区 │ │ │ │ L3: 字段权限 (Field Permission) [预留] │ │ └── 能否查看/修改某个字段 │ │ └── 例：只能看 salary 字段的前 3 位 │ │ │ └──────────────────────────────────────────────────────────┘

二、数据库表结构

2.1 核心表

-- ============================================================ -- 1. 权限表（核心） -- ============================================================ CREATE TABLE permissions ( id SERIAL PRIMARY KEY, permission_code VARCHAR(100) NOT NULL UNIQUE, -- 例：user:create permission_name VARCHAR(200) NOT NULL, -- 例：创建用户 permission_type VARCHAR(20) NOT NULL DEFAULT 'API', resource_type VARCHAR(50) NOT NULL, -- 例：user, workarea action VARCHAR(20) NOT NULL, -- create | read | update | delete description VARCHAR(500), is_system BOOLEAN DEFAULT false, -- 系统权限不可删除 created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); -- 索引 CREATE INDEX idx_permissions_resource ON permissions(resource_type); CREATE INDEX idx_permissions_action ON permissions(action); -- ============================================================ -- 2. 角色表 -- ============================================================ CREATE TABLE roles ( id SERIAL PRIMARY KEY, role_code VARCHAR(100) NOT NULL UNIQUE, -- 例：system_admin role_name VARCHAR(200) NOT NULL, -- 例：系统管理员 description VARCHAR(500), is_system BOOLEAN DEFAULT false, -- 系统角色不可删除 created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); -- ============================================================ -- 3. 角色 - 权限关联表 -- ============================================================ CREATE TABLE role_permissions ( id SERIAL PRIMARY KEY, role_id INTEGER NOT NULL REFERENCES roles(id) ON DELETE CASCADE, permission_id INTEGER NOT NULL REFERENCES permissions(id) ON DELETE CASCADE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, UNIQUE (role_id, permission_id) ); CREATE INDEX idx_role_permissions_role ON role_permissions(role_id); CREATE INDEX idx_role_permissions_perm ON role_permissions(permission_id); -- ============================================================ -- 4. 用户 - 角色关联表 -- ============================================================ CREATE TABLE user_roles ( id SERIAL PRIMARY KEY, user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE, role_id INTEGER NOT NULL REFERENCES roles(id) ON DELETE CASCADE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, UNIQUE (user_id, role_id) ); CREATE INDEX idx_user_roles_user ON user_roles(user_id); CREATE INDEX idx_user_roles_role ON user_roles(role_id);

2.2 数据权限表

-- ============================================================ -- 5. 用户 - 作业区关联表（数据权限） -- ============================================================ CREATE TABLE user_work_areas ( id SERIAL PRIMARY KEY, user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE, work_area_id INTEGER NOT NULL REFERENCES work_areas(id) ON DELETE CASCADE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, created_by INTEGER REFERENCES users(id), UNIQUE (user_id, work_area_id) ); CREATE INDEX idx_user_work_areas_user ON user_work_areas(user_id); CREATE INDEX idx_user_work_areas_area ON user_work_areas(work_area_id); -- ============================================================ -- 6. 权限配置审计日志表 -- ============================================================ CREATE TABLE permission_audit_log ( id BIGSERIAL PRIMARY KEY, operator_id INTEGER NOT NULL REFERENCES users(id), operator_name VARCHAR(200) NOT NULL, operation_type VARCHAR(50) NOT NULL, target_type VARCHAR(50) NOT NULL, target_id INTEGER NOT NULL, target_name VARCHAR(200), permission_ids_before TEXT, permission_ids_after TEXT, change_description VARCHAR(1000), ip_address VARCHAR(50), user_agent VARCHAR(500), created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); CREATE INDEX idx_permission_audit_operator ON permission_audit_log(operator_id); CREATE INDEX idx_permission_audit_target ON permission_audit_log(target_type, target_id); CREATE INDEX idx_permission_audit_time ON permission_audit_log(created_at);

2.3 菜单表（简化版）

-- ============================================================ -- 7. 菜单表（不再独立管理菜单权限） -- ============================================================ CREATE TABLE menus ( id SERIAL PRIMARY KEY, menu_code VARCHAR(100) NOT NULL UNIQUE, menu_name VARCHAR(200) NOT NULL, menu_path VARCHAR(500), parent_id INTEGER REFERENCES menus(id), sort_order INTEGER DEFAULT 0, icon VARCHAR(100), is_visible BOOLEAN DEFAULT true, required_permission VARCHAR(500), -- 逗号分隔的权限码 created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );

三、权限码规范

3.1 权限码格式

格式： {resource}:{action}
示例 说明
user:create 创建用户
user:read 查看用户
workarea:create 创建作业区
role:assign 分配角色

5.2 权限注解

// ApiPermission.java @Target(ElementType.METHOD) @Retention(RetentionPolicy.RUNTIME) @Documented public @interface ApiPermission { /** * 权限码，支持多个 */ String[] value(); /** * 逻辑关系：AND | OR */ Logical logical() default Logical.OR; enum Logical { AND, OR } } // 使用示例 @ApiPermission("user:create") @PostMapping("/create") public Result<User> createUser(@RequestBody UserCreateDTO dto) { ... } @ApiPermission(value = {"user:update", "user:delete"}, logical = Logical.OR) @DeleteMapping("/{id}") public Result<Void> deleteUser(@PathVariable Long id) { ... }

5.3 数据权限注解

// DataScope.java @Target(ElementType.METHOD) @Retention(RetentionPolicy.RUNTIME) @Documented public @interface DataScope { String tableAlias() default ""; ScopeType scopeType() default ScopeType.CUSTOM; enum ScopeType { ALL, // 全部数据（系统管理员） COMPANY, // 本公司数据 WORK_AREA, // 负责的作业区 SELF, // 仅本人 CUSTOM // 自定义 } } // 使用示例 @DataScope(tableAlias = "w", scopeType = ScopeType.WORK_AREA) @GetMapping("/list") public Result<List<WorkArea>> listWorkAreas() { ... }

六、前端实现方案

6.1 权限 Store

// stores/permission.js import { defineStore } from 'pinia' export const usePermissionStore = defineStore('permission', { state: () => ({ permissions: [], // 用户所有权限码 menuPermissions: [], // 菜单权限（从 API 权限推导） dataScopes: {} // 数据权限范围 }), getters: { hasPermission: (state) => (code) => { if (state.permissions.includes('*:*:*')) { return true // 系统管理员 } return state.permissions.includes(code) }, canViewMenu: (state) => (menuCode) => { const menuPermissionMap = { 'user_management': ['user:read'], 'role_management': ['role:read', 'perm:read'], 'workarea_list': ['workarea:read'], } const required = menuPermissionMap[menuCode] || [] return required.length === 0 || state.hasAnyPermission(required) } } })

6.2 权限指令

// directives/permission.js export const permission = { mounted(el, binding) { const permissionStore = usePermissionStore() const { value } = binding if (!checkPermission(permissionStore, value)) { el.parentNode?.removeChild(el) } } } // 使用示例 <el-button v-permission="'user:create'">创建用户</el-button> <el-button v-permission="['user:update', 'user:delete']">编辑/删除</el-button>

6.3 权限按钮组件

<!-- components/PermButton.vue --> <template> <el-button v-if="visible" v-bind="$attrs"> <slot /> </el-button> </template> <script setup> const props = defineProps({ permission: { type: [String, Array], required: true }, logical: { type: String, default: 'OR' } }) const visible = computed(() => { const codes = Array.isArray(props.permission) ? props.permission : [props.permission] return props.logical === 'AND' ? permissionStore.hasAllPermissions(codes) : permissionStore.hasAnyPermission(codes) }) </script>

八、权限缓存策略

8.1 Redis 缓存结构

# 用户权限缓存（2 小时过期） user:permissions:{userId} → Set<permission_code> # 用户角色缓存 user:roles:{userId} → Set<role_id> # 角色权限缓存（配置变更时失效） role:permissions:{roleId} → Set<permission_id> # 用户数据权限 user:dataScope:{userId} → Hash { scopeType, companyId, workAreaIds }

8.2 缓存失效策略

public void invalidatePermissionCache(Long userId, Long roleId) { // 失效用户权限缓存 redisTemplate.delete("user:permissions:" + userId); redisTemplate.delete("user:dataScope:" + userId); // 失效角色权限缓存 redisTemplate.delete("role:permissions:" + roleId); // 失效该角色下所有用户的权限缓存 Set<Long> userIds = findUsersByRole(roleId); for (Long uid : userIds) { redisTemplate.delete("user:permissions:" + uid); } }

十、安全注意事项

⚠️ 重要安全要求：
  • 后端是唯一验证点 - 前端权限检查仅用于 UI 展示，后端必须验证
  • 系统管理员特殊处理 - 系统管理员跳过所有权限检查
  • 权限变更需重新登录 - 权限修改后，用户需重新登录生效
  • 敏感操作审计日志 - 权限配置、角色变更等操作需记录审计日志
  • 防止权限提升 - 普通用户不能给自己或他人分配更高权限

十二、系统管理员操作指南

12.1 调整某个角色的权限

操作入口：系统管理 → 角色权限配置
  1. 选择要配置的角色
    • 加载该角色当前的权限配置
    • 显示所有权限（按资源类型分组）
    • 已选中的权限打勾
  2. 修改权限配置
    • 勾选/取消勾选权限
    • 支持全选/取消全选（按资源类型）
  3. 保存配置
    • 前端提交：roleId + permissionIds[]
    • 后端验证：当前用户是系统管理员
    • 删除旧的 role_permissions 记录
    • 插入新的 role_permissions 记录
    • 记录审计日志（permission_audit_log）
    • 保存历史版本（role_permission_history）
    • 失效缓存
  4. 生效
    • 用户下次请求时，从数据库重新加载权限
    • 或者用户重新登录

12.2 调整某个用户的权限

方式 A（推荐）：通过调整用户角色
  1. 进入用户管理 → 编辑用户 → 角色分配
  2. 勾选/取消勾选角色
  3. 保存
    • 删除旧的 user_roles 记录
    • 插入新的 user_roles 记录
    • 失效该用户的权限缓存
  4. 用户权限自动更新（继承新角色的权限）
方式 B：直接给用户分配权限（覆盖角色默认）

最终权限 = (角色权限并集 ∪ 用户 GRANT 权限) - 用户 DENY 权限

12.3 完整管理流程示例

# 场景 1：新创建一个角色并配置权限 步骤 1: 创建角色 POST /role/create { "roleCode": "safety_officer", "roleName": "安全员", "description": "负责作业区安全管理" } 步骤 2: 配置角色权限 PUT /permission/role/{roleId}/permissions { "permissionIds": [1, 2, 3, 15, 16, 20, 21, ...], "comment": "创建安全员角色，配置基础权限" } 步骤 3: 将角色分配给用户 POST /user/{userId}/roles { "roleIds": [23] } 步骤 4: 验证 - 使用该用户登录 - 检查返回的 permissions 列表 - 检查菜单显示 - 测试 API 访问

总结

关联关系总结

关系 表 说明
用户 → 角色 user_roles 一个用户可以有多个角色
角色 → 权限 role_permissions 一个角色可以有多个权限
用户 → 权限 计算得出 用户权限 = 所有角色权限的并集 ± 自定义权限
用户 → 数据范围 users.company_id + user_work_areas 公司维度 + 作业区维度

管理操作总结

操作 入口 影响范围 生效方式
新增/修改权限码 权限管理 所有角色 立即生效
调整角色权限 角色权限配置 该角色下所有用户 缓存失效后生效
调整用户角色 用户管理 → 编辑用户 单个用户 缓存失效后生效
给用户自定义权限 用户管理 → 自定义权限 单个用户 缓存失效后生效
