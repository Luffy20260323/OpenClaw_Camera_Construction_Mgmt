## 二、《场景故事_v5.0.md》理解

### 2.1 核心业务主线

```
项目创建 → 标段划分 → 点位生成 → 设备模型配置 → 
施工任务分配 → 现场数据采集 → 多级审核 → 
验收结算 → 运维管理
```

**业务特点**：
- 多角色协作（甲方、乙方、监理、系统管理员）
- 流程驱动（严格的审核流程）
- 数据追溯（所有操作留痕）
- 状态管理（点位状态、子任务状态）

### 2.2 关键创新点

#### 1️⃣ 三级审核体系 🎯

| 级别 | 审核人 | 审核内容 | 审核结论 |
|------|--------|----------|----------|
| 第一级 | 施工队长 | 子任务数据 | 通过/不通过 |
| 第二级 | 乙方项目管理 | 子任务级别 | 无风险通过/带风险通过/驳回 |
| 第三级 | 监理 | 点位级别 | 通过/不通过 |
| 第四级 | 甲方 | 点位级别 | 通过/不通过 |

**设计价值**：
- 责任明确，每级审核有清晰边界
- 带风险通过机制，平衡质量与进度
- 风险整改项闭环管理

#### 2️⃣ 子任务作废机制 ⚠️（v5.0 新增）

**触发场景**：
- 农户协调困难导致已完工子任务需重新执行
- 征地协议被推翻
- 规划变更

**作废流程**：
```
施工人员发起 → 施工队长审核 → 乙方项目管理审核 → 
监理审核 → 甲方审核 → 状态变更为"已报废"
```

**关键设计**：
- 联动作废：某子任务报废后，其后续所有已完成子任务联动作废
- 审计依据：作废子任务统计报表作为工程量审计核减依据

**设计价值**：
- 解决现场实际困难
- 工程量核算更准确
- 避免乙方虚报工程量

#### 3️⃣ 点位空壳记录 📍

**生命周期**：
```
标段创建 → 生成点位空壳（仅基础属性）→ 
指定设备模型 → 生成零部件实例 → 
关联属性模板 → 完整点位
```

**空壳属性**：
- 点位 ID
- 点位编码
- 所属标段
- 所属作业区

**为空属性**：
- 设备模型 ID
- 纬度、经度
- 地理空间坐标（geom）

**设计价值**：
- 延迟加载，避免过早占用存储空间
- 灵活支持不同设备模型配置

#### 4️⃣ 动态属性设计 🔧

**表结构**：
```sql
-- 零部件属性定义表（动态创建）
component_attr_def_{component_id}

-- 零部件属性值表（动态创建）
component_attr_value_{component_id}

-- 模板属性值表（动态创建）
component_template_value_{template_id}

-- 统一元数据管理
table_structure_registry
```

**存储策略**：
- 模板属性值：存储在 `component_template_value_{template_id}`
- 差异属性值：仅存储与模板不同的值在 `component_attr_value_{component_id}`

**设计价值**：
- 支持零部件种类灵活扩展
- 节省存储空间（模板复用 + 差异存储）
- 避免频繁修改表结构

#### 5️⃣ 项目沙盘可视化 🗺️

**点位状态（6 种）**：
| 状态 | 颜色 | 说明 |
|------|------|------|
| 未开始 | 黄色小圆点 | 所有子任务均未开始 |
| 施工中 | 橙色小圆点 | 任意子任务处于实施过程中 |
| 待审核 | 浅绿色小圆点 | 监理已通过并提交给甲方审核 |
| 整改中 | 浅红色小圆点 | 甲方审核不通过需要整改 |
| 审核通过 | 绿色小圆点 | 甲方审核后认可 |
| 已报废 | 灰色小圆点 | 有子任务被报废 |

**聚合显示**：
- 地图缩小时，多个点位叠加显示为带数值的黄色五角星
- 数值表示该位置叠加的点位数量
- 点击聚合图标可展开查看所有点位

**设计价值**：
- 直观展示项目整体进度
- 快速定位问题点位
- 支持多维度筛选（作业区、项目、标段）

### 2.3 角色权限设计

**公司类型**：
- 甲方公司（建设方）
- 乙方公司（施工方）
- 监理公司（监督方）
- 软件所有者公司（系统开发方）

**作业区数据隔离**：
- 甲方作业区角色只能查看和管理所属作业区的项目和点位
- 通过 `user_work_areas` 表实现用户与作业区关联
- 非作业区角色可查看全公司数据

## 三、《软件架构设计_v5.0.md》理解

### 3.1 技术选型分析

| 分层 | 技术选型 | 选型理由 |
|------|----------|----------|
| 后端 | Spring Boot 3.2 + Java 17 | 企业级成熟度最高，生态完善，强事务支持 |
| 前端 PC | Vue 3.4 + Element Plus | 国内生态好，学习曲线低，响应式设计 |
| 前端移动端 | Uni-app | 一套代码多端运行（iOS/Android/鸿蒙） |
| 数据库 | PostgreSQL 16 + PostGIS 3 | 开源免费，支持地理空间查询 |
| ORM | MyBatis-Plus 3.5 | 简化 CRUD，国内流行 |
| 缓存 | Redis 7 | 高性能缓存，支持分布式锁 |
| 文件存储 | MinIO/OSS | 开源对象存储，兼容 S3 协议 |
| 安全认证 | Spring Security + JWT | 成熟的安全框架 |

**技术选型对比**（后端框架）：

| 框架 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| Spring Boot | 企业级成熟、生态完善、性能优秀 | 开发效率较低、内存占用高 | 大型企业级项目 ✅ |
| NestJS | 开发效率高、技术栈统一 | 企业级经验较少 | 中小型项目 |
| FastAPI | 开发效率最高、自动文档 | 并发性能受限 | 快速原型/AI 项目 |

**选择 Spring Boot 的核心理由**：
1. 企业级项目，业务逻辑复杂
2. 数据一致性要求高（工程量统计、进度款、结算管理）
3. 权限体系复杂（RBAC + 作业区数据隔离）
4. 审计要求（所有操作留痕）
5. PostGIS 支持（Hibernate Spatial 对 PostGIS 支持最好）
6. 人才储备（国内 Java 开发者最多）

| component | 零部件管理 | 零部件种类、属性模板、动态表管理 |

- 零部件属性定义表：`component_attr_def_{component_id}`

### 3.5 安全设计

**认证机制**：JWT（JSON Web Token）

**Token 结构**：
```json
{
  "userId": 1,
  "username": "admin",
  "companyId": 1,
  "roles": ["system_admin"],
  "workAreaIds": [1, 2],
  "iat": 1710000000,
  "exp": 1710086400
}
```

**授权机制**：RBAC（Role-Based Access Control）

**权限粒度**：
- 菜单级权限（控制菜单显示）
- 按钮级权限（控制按钮显示）
- API 级权限（控制接口访问）
- 数据级权限（作业区数据隔离）

**数据安全**：
- 用户密码：BCrypt 加密
- 敏感信息：AES-256 加密（手机号、邮箱）
- 传输安全：HTTPS 协议（TLS 1.3）
- SQL 注入防护：MyBatis 参数化查询
- XSS 防护：输入过滤、输出转义
- CSRF 防护：Token 验证

### 3.6 性能优化

**数据库优化**：
- 合理设计索引（主键、外键、业务查询索引）
- 分页查询（避免一次性加载大量数据）
- 慢查询监控（PostgreSQL pg_stat_statements）

**缓存优化**：
- 热点数据缓存（用户信息、项目信息）
- 缓存预热（系统启动时加载基础数据）
- 缓存过期策略（TTL + 主动失效）

**异步处理**：
- 消息队列（RabbitMQ）解耦耗时操作
- 异步通知（邮件、短信）
- 异步报表生成（周报、结算报告）

## 五、我的疑问与建议

### 5.1 疑问 ❓

1. **消息队列的使用场景**
   - 文档提到 RabbitMQ，但具体哪些操作用异步？
   - 建议明确：通知推送、报表生成、批量审核

2. **移动端离线同步**
   - 施工人员可能在信号差的地方作业
   - 建议：本地 SQLite 缓存 + 冲突解决策略

3. **结算与财务对接**
   - 进度款申请后如何对接财务系统？
   - 建议：预留 API 接口或导出标准格式（如 Excel、PDF）

4. **动态表维护**
   - 零部件种类删除时，对应的动态表如何处理？
   - 建议：软删除 + 定期归档清理

5. **项目沙盘性能**
   - 大量点位（如 10000+）同时展示时的性能问题
   - 建议：点位状态数据预计算 + 分页加载 + 视口裁剪

### 5.2 建议 💡

1. **统一文件存储抽象层**
   - 开发用 MinIO，生产用 OSS
   - 建议：定义统一接口，避免代码耦合

2. **增加数据字典管理**
   - 系统运行时可配置的数据字典
   - 建议：字典表 + 缓存 + 管理界面

3. **完善日志体系**
   - 操作日志、审计日志、系统日志分离
   - 建议：ELK Stack 集中管理

4. **增加 API 版本管理**
   - 便于后续迭代和兼容
   - 建议：URL 版本号（/api/v1/...）

5. **考虑微服务拆分路径**
   - 当前是单体架构，但预留了拆分可能
   - 建议：模块间接口清晰，便于后续拆分
