# 软件研发日志命名规则

**版本：** v1.0  
**制定日期：** 2026 年 3 月 19 日  
**适用范围：** 视频监控点位施工项目管理系统研发团队

---

## 一、命名格式

```
日期_作者_文档类型_版本号.md
```

各部分使用下划线 `_` 连接，具体说明如下：

### 1. 日期（Date）
- **格式：** `YYYY-MM-DD`
- **示例：** `2026-03-19`
- **说明：** 文档创建日期，使用公历日期

### 2. 作者（Author）
- **格式：** 中文姓名或昵称
- **示例：** `柳生`、`张三`、`李四`
- **说明：** 文档作者的真实姓名或团队内昵称

### 3. 文档类型（DocType）
- **格式：** 简体中文，2-4 个字
- **可选值：**
  - `今日总结` - 每日开发日志
  - `开发日志` - 特定功能开发记录
  - `测试报告` - 测试执行报告
  - `会议纪要` - 团队会议记录
  - `技术调研` - 技术方案调研报告
  - `需求分析` - 需求分析文档
  - `设计文档` - 系统设计文档
  - `用户手册` - 面向用户的操作手册
  - `维护手册` - 面向运维的维护手册
  - `部署指南` - 系统部署指导文档

### 4. 版本号（Version）
- **格式：** `vX.Y` 或 `vX.Y.Z`
- **示例：** `v1.0`、`v1.1`、`v2.0`
- **说明：**
  - 当天第一次生成某类文档使用 `v1.0`
  - 当天第二次生成使用 `v2.0`，以此类推
  - 跨天重新从 `v1.0` 开始

```
2026-03-19_柳生_维护手册_v1.0.md
2026-03-19_柳生_部署指南_v1.0.md
```

## 三、版本递增规则

### 1. 同一天内
- 同一作者创建同一类型的文档，版本号递增
- 示例：
  ```
  2026-03-19_柳生_今日总结_v1.0.md  # 第一次
  2026-03-19_柳生_今日总结_v2.0.md  # 第二次（更新版本）
  ```

### 2. 不同作者
- 不同作者创建同一类型的文档，版本号各自独立
- 示例：
  ```
  2026-03-19_柳生_今日总结_v1.0.md
  2026-03-19_张三_今日总结_v1.0.md  # 不同作者，从 v1.0 开始
  ```

### 3. 跨天处理
- 新的一天重新开始，版本号从 `v1.0` 开始
- 示例：
  ```
  2026-03-19_柳生_今日总结_v3.0.md  # 3 月 19 日第 3 版
  2026-03-20_柳生_今日总结_v1.0.md  # 3 月 20 日第 1 版
  ```

## 五、文档内容规范

### 1. 开发日志（今日总结）
应包含以下内容：
- 开发内容总结
- 技术实现细节
- Git 提交记录
- 测试情况
- 待办事项
- 文档更新记录

### 2. 测试报告
应包含以下内容：
- 测试范围
- 测试环境
- 测试用例执行情况
- 缺陷统计
- 测试结论

### 3. 用户手册
应包含以下内容：
- 系统概述
- 功能说明
- 操作步骤
- 常见问题

### 4. 维护手册
应包含以下内容：
- 系统架构
- 部署说明
- 配置说明
- 故障排查
- 备份恢复

---

## 六、注意事项

1. **文件名使用中文**：便于团队成员快速识别文档内容
2. **日期格式统一**：使用 `YYYY-MM-DD` 格式，便于排序
3. **版本号清晰**：避免使用 `final`、`latest` 等模糊表述
4. **及时提交**：文档创建后及时 commit 并 push 到 Git 仓库
5. **避免覆盖**：更新文档时创建新版本，保留历史版本

## 七、工具支持

### Git 命令示例

```bash
# 查看当天已创建的文档
ls docs/$(date +%Y-%m-%d)*

# 创建新的开发日志
touch docs/$(date +%Y-%m-%d)_$(whoami)_今日总结_v1.0.md

# 查看文档版本历史
git log -- docs/2026-03-19_柳生_今日总结_v*.md
```
