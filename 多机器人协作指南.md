## 🎯 协作模式

### 推荐：分工协作模式

```
用户
 ├── dev-1 (开发 1) ← 后端任务
 ├── dev-2 (开发 2) ← 前端任务
 ├── test-1 (测试 1) ← 测试任务
 └── doc-1 (文档 1) ← 文档任务
```

### 工作流程

1. **任务分配**
   - 用户在对应机器人的 1v1 对话中分配任务
   - 或在与 mgr-1 的对话中统一分配

2. **任务执行**
   - 各机器人独立执行任务
   - 完成后更新任务板

3. **进度同步**
   - 通过 TASK_BOARD.md 同步状态
   - 通过 memory/ 文件记录日志

4. **代码合并**
   - 使用 Git 分支管理
   - 定期合并到 master

## 📝 任务板使用

### 文件位置

`/root/.openclaw/workspace/TASK_BOARD.md`

### 更新规范

#### 开始任务

```markdown
## 📋 进行中

| 任务 | 负责人 | 机器人 | 状态 | 更新时间 |
|------|--------|--------|------|----------|
| 功能名称 | 姓名 | dev-X | 🔄 进行中 | YYYY-MM-DD HH:mm |
```

#### 完成任务

```markdown
1. 从"进行中"移除
2. 添加到"已完成"
3. 填写完成时间
```

#### 示例

```markdown
## ✅ 已完成

| 任务 | 完成时间 | 负责人 | 机器人 |
|------|----------|--------|--------|
| 文档中心 API | 2026-03-30 08:57 | 柳生 | dev-1 |
```

## 🔧 Git 协作流程

### 分支策略

```
master (生产分支)
├── feature/dev1-backend-xxx (dev-1 开发)
├── feature/dev2-frontend-yyy (dev-2 开发)
└── feature/test1-zzz (test-1 开发)
```

### 开发流程

```bash
# 1. 创建功能分支
git checkout -b feature/your-feature

# 2. 开发并提交
git add .
git commit -m "feat: 完成功能描述"

# 3. 推送到远程
git push origin feature/your-feature

# 4. 创建 Pull Request
# (通过 GitHub 网页或 gh CLI)

# 5. 代码审查后合并
```

### 提交规范

```
feat: 新功能
fix: 修复 bug
docs: 文档更新
style: 代码格式
refactor: 重构
test: 测试
chore: 构建/工具
```

## ⚠️ 注意事项

### 避免冲突

1. **文件锁**
   - 编辑同一文件前先检查
   - 使用 Git 分支避免直接冲突

2. **任务分配**
   - 明确任务边界
   - 避免多人修改同一模块

3. **提交频率**
   - 小步提交，频繁推送
   - 避免大提交难以审查

### 信息同步

1. **及时更新任务板**
2. **每日记录工作日志**
3. **重要决策写入 MEMORY.md**

---

## 📊 最佳实践

### ✅ 推荐做法

1. **任务粒度适中**
   - 单个任务 2-4 小时完成
   - 便于跟踪和协调

2. **及时同步状态**
   - 开始任务时更新
   - 完成任务时更新
   - 遇到问题时更新

3. **代码审查**
   - PR 需要其他机器人审查
   - 使用 GitHub Review 功能

4. **文档同步**
   - 代码完成后更新文档
   - API 变更及时通知

### ❌ 避免做法

1. **长时间不更新状态**
2. **大提交不说明**
3. **直接修改 master**
4. **忽略测试**

## 🎯 示例场景

### 场景 1：新功能开发

```
用户 → dev-1: 开发文档中心 API
dev-1 → TASK_BOARD: 更新状态为"进行中"
dev-1 → 开发 → 测试 → 部署
dev-1 → TASK_BOARD: 更新状态为"完成"

用户 → dev-2: 开发文档中心前端
dev-2 → TASK_BOARD: 查看 API 已完成
dev-2 → 开发前端 → 测试
dev-2 → TASK_BOARD: 更新状态为"完成"
```

### 场景 2：Bug 修复

```
用户 → test-1: 发现文档中心页面 bug
test-1 → TASK_BOARD: 创建 bug 任务
test-1 → dev-2: @dev-2 前端有 bug 需要修复
dev-2 → 修复 bug → 提交
dev-2 → test-1: 已修复，请验证
test-1 → 验证 → TASK_BOARD: 更新为"完成"
```

## 📞 故障排除

### Q: 机器人不知道其他人在做什么？

A: 查看 TASK_BOARD.md 和 memory/ 文件

### Q: 代码冲突怎么办？

A: 
1. git pull 最新代码
2. 手动解决冲突
3. git add + git commit
4. 继续推送

### Q: 任务优先级冲突？

A: 通过 mgr-1 协调，或直接在 TASK_BOARD 中讨论
